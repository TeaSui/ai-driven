# Next Phase Roadmap

This document outlines improvements and new features to pursue after the initial refactoring (legacy removal, CDK cleanup, documentation overhaul) is complete.

## Priority 1 — Reliability & Observability

### 1.1 Structured Logging & Tracing
- Add correlation IDs (ticket key + execution ID) to all log entries via MDC
- Enable AWS X-Ray tracing on all Lambda functions
- Create CloudWatch dashboards for workflow success/failure rates, duration percentiles, and AI response quality metrics

### 1.2 Configuration Externalization
Harcoded magic numbers should become environment variables:
| Current Value | Location | Config Name |
|---------------|----------|-------------|
| 100KB | `BitbucketFetchHandler` | `MAX_FILE_SIZE_BYTES` |
| 700,000 chars | `BitbucketFetchHandler` | `MAX_CONTEXT_CHARS` |
| 3MB | `BitbucketFetchHandler` | `MAX_ARCHIVE_SIZE_MB` |
| 8,192 tokens | `ClaudeInvokeHandler` | `CLAUDE_MAX_TOKENS` |
| 0.2 | `ClaudeInvokeHandler` | `CLAUDE_TEMPERATURE` |
| 7 days | CDK stack | `MERGE_WAIT_TIMEOUT_DAYS` |

### 1.3 Dependency Injection
Replace no-arg constructors with constructor DI (e.g., Dagger 2) for testability. Current pattern:
```java
// Current: hard to test
public FetchTicketHandler() {
    this.ticketStateRepository = new TicketStateRepository(DynamoDbClient.create(), System.getenv("DYNAMODB_TABLE_NAME"));
}

// Target: testable
public FetchTicketHandler(TicketStateRepository repo, SecretsService secrets, String jiraSecretArn) {
    this.ticketStateRepository = repo;
    this.secretsService = secrets;
    this.jiraSecretArn = jiraSecretArn;
}
```

## Priority 2 — AI Quality

### 2.1 Incremental Context
Instead of downloading the entire repo every time, implement:
1. Keyword extraction from ticket → targeted file search via Bitbucket API
2. File tree + relevant files only → reduces Claude token usage by ~80%
3. Fall back to full-repo download for ambiguous tickets

### 2.2 Multi-Model Support
Allow choosing Claude model per ticket via labels:
- `ai-model:haiku` → Fast/cheap for small changes
- `ai-model:sonnet` → Default balanced option
- `ai-model:opus` → Complex architectural changes

### 2.3 Prompt Iteration Feedback Loop
Track AI generation quality:
- Record PR approval/rejection rates per ticket type
- A/B test prompt variations
- Store successful prompts as templates

---

## Priority 3 — Platform Expansion

### 3.1 GitHub Support
Add GitHub as an alternative to Bitbucket:
- New `github-client` module (similar to `bitbucket-client`)
- Auto-detect platform from Jira ticket metadata (custom field or label)
- Shared interface: `SourceControlClient` with `BitbucketClient` and `GitHubClient` implementations

### 3.2 Multi-Repository Support
Current: one hardcoded workspace/repo pair
Target: resolve target repo from Jira ticket metadata:
- Custom Jira field: `Repository URL`
- Label convention: `repo:my-service`

### 3.3 Linear / Notion / Shortcut Integration
Abstract the issue tracker integration:
- `IssueTrackerClient` interface
- `JiraClient`, `LinearClient`, `NotionClient` implementations
- Webhook adapters per platform

---

## Priority 4 — Cost & Security

### 4.1 Cost Controls
- CloudWatch billing alarms for Claude API spend
- Monthly budget limits with automatic workflow disabling
- Per-ticket cost tracking (tokens used, API calls made)

### 4.2 Audit Trail
- Log all AI-generated code diffs to a dedicated S3 bucket
- Record prompt + response pairs for compliance
- Add a "generated by AI" footer to all PRs

### 4.3 Input Sanitization
- Validate and sanitize Jira ticket content before prompt injection
- Detect and reject tickets with potentially malicious instructions
- Rate limit per-user webhook triggers
